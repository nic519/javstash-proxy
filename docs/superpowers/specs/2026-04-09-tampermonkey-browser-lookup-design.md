# 油猴站点内番号查询设计

## 背景

当前 `javstash-proxy` 已经具备：

- 代理 JavStash 上游查询
- 基于缓存返回已有结果
- 对标题和简介执行翻译

现在需要增加一个油猴插件场景：当用户访问第三方站点的详情页时，脚本自动识别页面中的番号，并在番号旁边提供一个查询入口。用户点击后，直接在当前页面显示由 `javstash-proxy` 返回的标题和简介。

用户会自行提供 JavStash `ApiKey`。一期只做只读查询，不做标签写入。

## 目标

让用户在支持的网站详情页中：

- 自动识别当前页面番号
- 在番号旁边看到查询按钮
- 配置一次 JavStash `ApiKey`
- 点击按钮后，通过 `javstash-proxy` 获取该番号的标题和简介
- 在页内浮层中查看结果

## 非目标

- 不实现标签读写
- 不实现多站点批量适配
- 不做列表页批量注入
- 不直接让油猴脚本调用上游 GraphQL
- 不在一期支持可配置的 Proxy 地址

## 总体方案

整体链路为：

`第三方站点页面 -> 油猴脚本识别番号 -> 请求 javstash-proxy 浏览器查询接口 -> 代理查询/缓存/翻译 -> 返回精简结果 -> 浮层展示`

职责划分：

- 油猴脚本：页面识别、配置管理、按钮注入、浮层展示
- `javstash-proxy`：接收浏览器查询请求、校验参数、查询上游、读取缓存、翻译并返回稳定 JSON

这样可以保证：

- 油猴脚本尽量轻量
- GraphQL 和上游细节不暴露给脚本
- 后续增加标签能力时，仍可复用相同浏览器接口体系

## 一期范围

一期限定为：

- 支持 `javbus` 详情页
- 支持从 URL 或页面文本中提取单个番号
- 油猴菜单中提供 `ApiKey` 设置入口
- 按钮点击后弹出当前页浮层
- 浮层展示标题、简介、加载态、失败态、未配置态

## 浏览器接口设计

新增只读接口：

- `GET /api/browser/lookup?code=SSIS-828`

请求头：

- `X-Javstash-Api-Key: <user api key>`

请求要求：

- `code` 必填
- `code` 在服务端做标准化处理，如大写、去除多余空格
- `ApiKey` 必须通过请求头传入，不通过 URL 查询参数传入

成功响应示例：

```json
{
  "ok": true,
  "code": "SSIS-828",
  "title": "中文标题",
  "description": "中文简介",
  "translated": true,
  "source": "javstash-proxy"
}
```

失败响应示例：

```json
{
  "ok": false,
  "error": "NOT_FOUND",
  "message": "未找到对应条目"
}
```

建议错误码：

- `INVALID_CODE`
- `MISSING_API_KEY`
- `NOT_FOUND`
- `UPSTREAM_ERROR`
- `INTERNAL_ERROR`

## 服务端处理流程

浏览器接口内部流程：

1. 读取 `code` 与 `X-Javstash-Api-Key`
2. 校验 `code` 和 `ApiKey` 是否存在
3. 将 `code` 归一化
4. 基于内部查询逻辑向上游发起请求
5. 优先复用现有缓存、翻译与响应处理能力
6. 从结果中提取油猴场景所需的最小字段
7. 返回稳定、精简的 JSON 响应

接口不应把原始 GraphQL 结构直接透出给油猴脚本。

## 油猴脚本结构

建议将脚本划分为 4 个部分：

- `config`
  - 管理本地配置
  - 一期只保存 `ApiKey`

- `site`
  - 判断当前是否为支持的页面
  - 识别并归一化番号
  - 找到按钮注入位置

- `api`
  - 调用 `javstash-proxy` 浏览器接口
  - 统一处理超时、错误码和空结果

- `ui`
  - 渲染按钮
  - 渲染浮层
  - 渲染未配置、加载中、成功、失败状态

## 页面识别与注入策略

一期仅支持 `javbus` 详情页。

推荐识别顺序：

1. 优先从 URL 提取番号
2. 若 URL 无法提取，则从页面中显示番号的文本区域提取
3. 如果仍无法确定，脚本不注入按钮

按钮注入策略：

- 将按钮插入到页面中的番号文本附近
- 若未找到稳定的注入点，则允许退化为插入在标题区或信息区的末尾
- 同一页面只允许注入一次

## 配置管理

油猴菜单提供：

- `设置 JavStash ApiKey`
- `清除 JavStash ApiKey`

一期不提供 Proxy 地址设置，脚本内写死代理域名。

用户点击按钮但尚未配置 `ApiKey` 时：

- 不弹输入框
- 直接展示未配置态浮层
- 引导用户通过油猴菜单完成设置

这样可以把查询流程和配置流程解耦，便于后续增加其他配置项。

## 浮层交互

浮层为页内只读信息面板，推荐固定在页面右下角或右侧中部。

浮层需要支持：

- 打开
- 关闭
- 加载中提示
- 查询成功展示标题和简介
- 未找到提示
- 请求失败提示

浮层不承载编辑操作，也不承担配置录入。

## 样式隔离

由于目标站点样式不可控，需要避免脚本 UI 被宿主站覆盖。

控制方式：

- 使用独立 class 前缀
- 避免依赖宿主站样式
- 若普通样式隔离不够，再升级为 Shadow DOM

一期可先用独立 class 前缀实现，只有在样式冲突明显时再引入 Shadow DOM。

## 错误处理

服务端：

- 缺少 `code`：返回 `400`
- 缺少 `ApiKey`：返回 `401` 或 `400`
- 上游查询失败：返回 `502` 或映射后的统一错误
- 未命中数据：返回 `404` 或 `200 + ok: false`

油猴端：

- 未配置 `ApiKey`：显示未配置提示
- 请求超时：显示超时错误
- 服务端错误：显示查询失败提示
- 未找到结果：显示未找到提示

## 安全与合理性

一期允许用户在本地保存 `ApiKey`，这是可接受的工程折中，但不应被视为高安全方案。

约束：

- `ApiKey` 仅保存在用户浏览器侧脚本存储中
- 请求时通过 Header 发送，不拼接到 URL
- 浮层和日志中都不回显 `ApiKey`

## 扩展位

虽然一期不实现标签功能，但需要为后续扩展预留边界：

- 浏览器接口路径保持独立，不复用通用 GraphQL 入口
- 油猴脚本保留 `config/api/ui` 分层
- 浮层未来可扩展为详情 + 标签操作面板

这样二期增加“标签读写”时，可以在不推翻当前查询能力的前提下继续演进。

## 测试策略

服务端建议覆盖：

- `code` 参数校验
- `ApiKey` 缺失校验
- 查询成功的精简响应格式
- 上游失败与未找到的错误映射

油猴脚本建议覆盖：

- `javbus` 页面番号提取
- 单次按钮注入
- 未配置 `ApiKey` 时的提示
- 查询成功后的浮层展示

## 风险与控制

- `javbus` 页面结构可能变化，因此注入点选择需要允许降级
- 番号格式可能存在大小写、连字符和前导零差异，因此服务端和脚本都需要归一化
- 若宿主站点样式污染浮层，需要升级样式隔离策略
- 一期写死 Proxy 域名，后续若部署域名变化，需要更新脚本
